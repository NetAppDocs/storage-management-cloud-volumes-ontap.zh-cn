= Cloud Volumes ONTAP HA 配置
:allow-uri-read: 


[role="lead"]
Cloud Volumes ONTAP 中的高可用性 (HA) 配置可确保企业存储的业务连续性和容错能力。通过在独立的故障域中部署两个 ONTAP 节点，系统可以在发生硬件故障、网络中断或日常维护时提供无中断操作 (NDO) 和无缝故障转移。

当您的架构需要以下条件时，部署 HA 对：

* *关键任务工作负载*：无法容忍停机的数据库和企业应用程序。
* *无中断升级*：升级到 ONTAP 版本或更改底层云计算实例，而不使存储脱机。
* *多区域弹性*：针对云区域内的孤立可用区故障提供保护。




[[how-ha-pairs-work]]
== HA对的工作原理

从管理角度来看，HA 配置作为主动-主动集群运行，而单个数据聚合由特定节点所有。

* *同步复制*：写入主节点 NVRAM 的数据在向客户端确认写入之前，会同步镜像到伙伴节点 NVRAM。
* *Mediator 实例支持*：第三方 Mediator 实例监控两个节点的运行状况。如果主节点变得无响应，Mediator 将协调存储接管。
* *存储接管*：幸存节点接管合作伙伴的存储聚合和网络接口的所有权，在几秒钟内恢复数据服务。




[[ha-configuration-requirements-and-limitations]]
== HA 配置要求和限制

在 Cloud Volumes ONTAP 中部署 HA 对时，请考虑以下要求和限制：

* *计算大小*：HA 对中的两个节点必须使用完全相同的云计算实例类型或机器形状。
* *协议支持*：Hyper-V 和 SQL Server 工作负载支持 SMB/CIFS 持续可用性 (CA)，但需要特定的 ONTAP 配置标志。
* *故障转移时间*：虽然存储接管通常以秒为单位，但必须正确配置客户端超时设置（如 iSCSI 启动程序超时），以防止应用程序中断。




[[operations-unavailable-when-a-node-in-an-ha-pair-is-offline]]
== HA 对中的节点脱机时操作不可用

当 HA 对中的一个节点不可用时，另一个节点将为其伙伴提供数据以提供持续的数据服务。这被称为_存储接管_。在存储恢复完成之前，有些操作无法执行。


TIP: 当 HA 对中的某个节点不可用时， NetApp Console中的系统状态为“Degraded”。

存储接管后无法执行以下操作：

* 支持注册
* 许可证变更
* 实例或虚拟机类型更改
* 写入速度变化
* CIFS 设置
* 更改配置备份的位置
* 设置集群密码
* 管理磁盘和聚合（高级分配）


存储交还完成且系统状态恢复正常后，这些操作将再次可用。



[[storage-vm-disaster-recovery-for-cloud-volumes-ontap]]
== Storage VM 灾难恢复（适用于 Cloud Volumes ONTAP）

NetApp Console 不为存储虚拟机 (SVM) 灾难恢复 (DR) 提供设置或业务流程支持。要为 Cloud Volumes ONTAP 设置 DR，您必须使用 ONTAP System Manager 或 ONTAP CLI 来配置 SnapMirror SVM 复制。

如果在两个 Cloud Volumes ONTAP 系统之间设置 SnapMirror SVM 复制，则复制必须在两个 HA 对系统或两个单节点系统之间进行。无法在 HA 对和单节点系统之间设置 SnapMirror SVM 复制。

有关ONTAP CLI 说明，请参阅以下文档。

* https://library.netapp.com/ecm/ecm_get_file/ECMLP2839856["《SVM 灾难恢复准备快速指南》"^]
* https://library.netapp.com/ecm/ecm_get_file/ECMLP2839857["《SVM 灾难恢复快速指南》"^]
